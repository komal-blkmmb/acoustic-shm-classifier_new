// UPLOAD ITERATION DATA
Upload the ZIP file(s) from each recording iteration. Each ZIP contains 27 audio files (one per cell), named like 1a.m4a, 2g.m4a, etc. You don't need all 5 — even 1 iteration will work. Once uploaded, the app automatically detects each hit using energy thresholding and extracts 154 audio features (128 PSD bins + 26 MFCC values) from every hit. STEP 03 / 06
// INSPECT EXTRACTED HITS
Before training, verify that the hit extraction worked correctly. Select a cell and iteration to see all hits overlaid on the same time axis. Good extraction = all hits look similar (same shape, same duration). Red-bordered hits differ significantly from the average — they may be echo artefacts or noise triggers. The mean ± 1 std band shows how consistent the hits are.
STEP 04 / 06
// TRAIN CLASSIFIERS
Select which models to train. The dataset is shuffled and split 70/30 — 70% for training, 30% for validation. Because both sets come from the same recording sessions, validation accuracy will be high but optimistic (the true unseen test happens in Step 5).

KNN finds the 5 most similar training hits and votes. Decision Tree learns if-then rules on features. Logistic Regression fits a linear probability boundary. SVM finds the widest possible margin between classes.

Now we test whether models generalize to new data collected in a different session (your HW2 dataset). Upload the HW2 ZIP — files should be named like 1a_hit1.wav, 2g_hit3.wav. The models are used exactly as trained — no retraining. A big accuracy drop from Step 4 means the model learned session-specific patterns rather than true structural damage signatures. STEP 06 / 06
// CLASSIFY A SINGLE HIT
Upload any single percussion hit (.wav or .m4a) and all trained models will classify it instantly. The same feature extraction pipeline runs: RMS normalize → extract 128 PSD bins + 26 MFCC values → StandardScaler normalize → classify. This is what deployment would look like — tap a cell, record it, get a health verdict in seconds.
